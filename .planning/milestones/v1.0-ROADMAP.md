# Roadmap: GSD-R

## Overview

GSD-R transforms GSD's code-commit workflow into a research-note workflow. The roadmap follows the architecture's dependency graph: fork and scaffold the project structure with templates, then build the atomic vault write operation, then layer source acquisition on top of it, then adapt the agent/planner layer, then wire verification and workflows together, and finally validate the whole system end-to-end with output formats and a real research project.

### Phase 2: Vault Write and State
**Goal**: Research notes are written atomically -- note file, sources folder, SOURCE-LOG.md, and git commit all succeed or all roll back -- and STATE.md tracks note status
**Depends on**: Phase 1
**Requirements**: FOUN-05, KNOW-02, KNOW-03
**Success Criteria** (what must be TRUE):
  1. A vault write operation creates a note file, its `-sources/` folder, a SOURCE-LOG.md inside it, and a git commit in a single atomic step
  2. If any part of the vault write fails (e.g., git commit fails), no partial artifacts remain on disk
  3. STATE.md displays a note-status table showing each note's status (draft/reviewed/final/source-incomplete) and a source-gap table listing missing sources with reason and impact

### Phase 3: Source Acquisition
**Goal**: Every cited source in a research note has a corresponding local file acquired through a reliable fallback chain, with full audit trail
**Depends on**: Phase 2
**Requirements**: SRC-01, SRC-02, SRC-03, SRC-04, SRC-05, SRC-06, SRC-07
**Success Criteria** (what must be TRUE):
  1. Given a URL, the acquisition function tries firecrawl, then web_fetch, then wget/curl, and falls back to marking unavailable -- producing a local file or a documented gap, never silently failing
  2. SOURCE-LOG.md for each note lists every acquisition attempt with source name, URL, method used, local file path, status (full/partial/paywall/unavailable), and notes
  3. Source files follow the naming convention `{descriptive-slug}_{date-acquired}.{ext}` and are saved in original format (PDFs as PDFs, HTML as markdown, screenshots as PNG)
  4. Every claim in a note's Analysis section references a specific local file in `-sources/` (not a URL), and synthesis notes trace back through individual notes to source files (two-hop verifiability)
  5. Unavailable sources are documented in SOURCE-LOG.md and flagged in the note's Open Questions section but do not block task completion
**Plans**: 2 plans

Plans:
- [x] 03-01-PLAN.md -- Source acquisition engine with fallback chain (TDD)
- [x] 03-02-PLAN.md -- Content validation and finding-to-source traceability

### Phase 4: Research Agents and Planner
**Goal**: Research tasks are planned with source-aware XML templates and executed by specialized researcher subagents, with the plan-checker enforcing source discipline
**Depends on**: Phase 3
**Requirements**: ORCH-04, ORCH-05, ORCH-06, ORCH-07, KNOW-01
**Success Criteria** (what must be TRUE):
  1. Four researcher agent prompts exist (source/methods/architecture/limitations) and each produces research notes when spawned with a task
  2. The planner generates research task XML with `<src method="" format="">` blocks specifying acquisition method and expected format for each source
  3. The plan-checker rejects plans that duplicate BOOTSTRAP.md findings, exceed 3 sources per task, omit acquisition methods, or assign >30-page papers as secondary sources in multi-source tasks
  4. BOOTSTRAP.md is generated during new-project with three tiers (Established / Partially Established / Not Yet Researched) and is consulted by the plan-checker

### Phase 5: Verification and Workflows
**Goal**: The full discuss/plan/execute/verify loop works for research workflows, with two-tier verification ensuring both research quality and source completeness
**Depends on**: Phase 4
**Requirements**: ORCH-01, ORCH-02, ORCH-03, VERI-01, VERI-02, VERI-03, VERI-04
**Success Criteria** (what must be TRUE):
  1. The discuss/plan/execute/verify loop runs end-to-end for a research task: question is discussed, plan is generated with source specs, executor produces note + sources, verifier checks both quality and sources
  2. Tier 1 verification (goal-backward) checks "What must be TRUE for this research question to be answered?" and fails tasks that don't answer the question
  3. Tier 2 verification (source audit) confirms every reference has a file in `-sources/`, SOURCE-LOG.md accounts for all sources, and no orphan files exist in the sources folder
  4. Verification failures automatically generate fix tasks via `/gsd-r:quick`
  5. Independent research tasks execute in parallel waves with fresh 200K subagent contexts per task
**Plans**: 2 plans
