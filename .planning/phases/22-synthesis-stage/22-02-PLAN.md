**Workflow structure:**

```xml
<purpose>
Transform verified research notes into structured scholarship through thematic synthesis, theoretical integration, gap analysis, and argument construction. Generates a synthesis plan and delegates to execute-phase machinery.
</purpose>

<required_reading>
Read STATE.md and config before any operation.
</required_reading>

<process>
```

**Step 1: initialize**
- Read config to get `config.workflow.synthesis` value, `vault_path`, `review_type`, `researcher_tier`
- If `config.workflow.synthesis` is `false`: inform user synthesis is disabled for this review type, offer to enable via `/grd:settings`, exit
- Parse `$ARGUMENTS` for flags: `--partial`, `--skip-themes`, `--skip-framework`, `--skip-gaps`

**Step 2: validate_readiness**
- Find all investigation phases in ROADMAP.md (phases with plans that are NOT synthesis)
- For each phase, check verification status:
  - `passed` or `human_needed` -> OK (D-05: respect researcher's choice to proceed past human_needed)
  - `gaps_found` -> BLOCK (unless `--partial` flag)
  - Missing verification -> BLOCK (unless `--partial`)
- If blocked: present specific next actions (D-08):
  - Unverified phase -> "Run `/grd:verify-inquiry {phase}`"
  - No notes found -> "Run `/grd:conduct-inquiry` first"
- Collect all verified research notes from vault:
  - Walk `{vault_path}/` directory structure
  - Collect all `.md` files NOT in `-sources/` subdirectories and NOT `00-` prefixed
  - These are the notes to pass to synthesis agents
- If zero notes found: error with "No verified research notes found. Run `/grd:conduct-inquiry` first."

**Step 3: trap_04_synthesis_scope (TRAP-04 interactive gate)**

```
AskUserQuestion(
  header: "Synthesis Scope",
  question: "How would you like to synthesize your research?",
  options: [
    "Full synthesis (all 4 activities)" -- "Themes, framework, gaps, and argument construction. Recommended for systematic, integrative, and critical reviews.",
    "Themes + argument only" -- "Skip framework integration and gap analysis. Produces thematic summary and executive summary.",
    "Skip synthesis" -- "Keep verified notes without synthesis. You can run /grd:synthesize later."
  ]
)
```

If "Skip synthesis": exit with message "Synthesis skipped. Run `/grd:synthesize` when ready."
If "Themes + argument only": set internal flags to skip framework and gaps.

**Step 4: validate_skip_flags**
- Process skip flags: `--skip-themes`, `--skip-framework`, `--skip-gaps`
- If `--skip-themes`: check if `{vault_path}/00-THEMES.md` already exists. If not, error: "Cannot skip themes -- 00-THEMES.md does not exist and is required by framework, gaps, and argument."
- If `--skip-framework`: check if `{vault_path}/00-FRAMEWORK.md` already exists OR if framework is being generated in this run. If gaps depend on it and it won't exist, error: "Cannot skip framework -- 00-FRAMEWORK.md does not exist and is required by gap analysis (D-07)."
- If `--skip-gaps`: no dependency issue (argument can work without GAPS.md, it just omits that section)
- Combine TRAP-04 selections with CLI flags

**Step 5: validate_activity_prerequisites (D-07)**
Per-activity prerequisite validation. D-07 defines the full dependency chain:
- Themes: at least 1 verified note (already checked in step 2)
- Framework: requires THEMES.md + Theoretical Framework Survey note from Stage 1 researchers (search for notes from `grd-architecture-researcher` via inquiry subdirectory naming or note content). If missing and not skipped: warn "Missing theoretical framework survey. Check Stage 1 researcher output or use --skip-framework."
- Gaps: requires THEMES.md + FRAMEWORK.md (per D-07: "Gaps requires THEMES.md + FRAMEWORK.md"). If --skip-framework and no existing FRAMEWORK.md: error "Cannot run gap analysis without FRAMEWORK.md. Use --skip-gaps or generate framework first."
- Argument: requires at minimum THEMES.md (framework and gaps are optional based on skip flags)

**Step 6: generate_synthesis_plan**
Generate a PLAN.md file at `.planning/phases/synthesis/synthesis-PLAN.md` (or use the study's planning structure).

The generated plan has tasks organized in **4 waves** (per D-07 dependency chain: gaps requires FRAMEWORK.md, so framework and gaps cannot run in parallel):

Wave 1 (unless --skip-themes):
- Task: Thematic Synthesis
  - Agent type: `grd-thematic-synthesizer`
  - Input files: all verified note paths (from step 2)
  - Output: `{vault_path}/00-THEMES.md`

Wave 2 (unless --skip-framework or "Themes + argument only"):
- Task: Theoretical Integration
  - Agent type: `grd-framework-integrator`
  - Input files: `{vault_path}/00-THEMES.md` + Theoretical Framework Survey note + all verified notes
  - Output: `{vault_path}/00-FRAMEWORK.md`

Wave 3 (unless --skip-gaps or "Themes + argument only"):
- Task: Gap Analysis
  - Agent type: `grd-gap-analyzer`
  - Input files: `{vault_path}/00-THEMES.md` + `{vault_path}/00-FRAMEWORK.md` + all verified notes
  - Output: `{vault_path}/00-GAPS.md`

Wave 4 (always runs if synthesis enabled):
- Task: Argument Construction
  - Agent type: `grd-argument-constructor`
  - Input files: `{vault_path}/00-THEMES.md` + (if exists) `{vault_path}/00-FRAMEWORK.md` + (if exists) `{vault_path}/00-GAPS.md` + `.planning/PROJECT.md` (for deliverable_format)
  - Output: `{vault_path}/00-Executive-Summary.md`

**Step 7: execute_synthesis**
Delegate the generated plan to execute-phase machinery:
- Use the same wave execution pattern as conduct-inquiry.md
- Each task spawns a subagent with the agent type specified in the task
- Agent prompts loaded from `grd/agents/{agent-type}.md`
- Pass file paths in `<files_to_read>` block of the spawned agent prompt
- Git commit after each wave completion

**Step 8: report_completion**
Report synthesis results:
```
Synthesis complete:
- Themes: {vault_path}/00-THEMES.md ({N} themes identified)
- Framework: {vault_path}/00-FRAMEWORK.md (if generated)
- Gaps: {vault_path}/00-GAPS.md ({N} gaps identified) (if generated)
- Executive Summary: {vault_path}/00-Executive-Summary.md

Next: /grd:complete-study
```
